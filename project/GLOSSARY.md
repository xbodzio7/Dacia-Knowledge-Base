\# Project Glossary



\*\*Project:\*\* Dacia Knowledge Base (DKB)



\*\*Version:\*\* 1.0



\*\*Status:\*\* Active



\---



\# Purpose



This document defines the terminology used throughout the project.



Every important project document should use these terms consistently.



\---



\# AI



The AI assistant collaborating on the project.



Responsible for analysis, implementation proposals, documentation, validation and architectural recommendations.



\---



\# Architecture Decision



A decision that affects the long-term structure of the project.



Architecture decisions are recorded in DECISIONS.md.



\---



\# Attribute



A single measurable or descriptive property of a vehicle.



Examples:



\- Engine displacement

\- Fuel type

\- Number of doors



\---



\# Backlog



A prioritised list of future work.



Maintained in ROADMAP.md.



\---



\# Business Rule



A rule describing valid relationships between project data.



Business rules are validated only after repository integrity has been verified.



\---



\# Commit



A single logical repository change.



Each commit should have one primary responsibility.



\---



\# Data Model



The logical structure of the Knowledge Base.



CSV files are only one physical representation of the model.



\---



\# Dataset



A collection of related project data.



Examples:



\- models

\- engines

\- gearboxes



\---



\# Documentation Sprint



A sprint dedicated primarily to documentation improvements.



No functional implementation is required.



\---



\# DKB



Abbreviation for Dacia Knowledge Base.



\---



\# Enum



A controlled list of predefined values.



Example:



Fuel Type



\- Petrol

\- Diesel

\- LPG

\- Hybrid



\---



\# Knowledge Base



The complete structured representation of project knowledge.



Includes:



\- documentation

\- master data

\- relationships

\- validation rules



\---



\# Master Data



Core reference data shared across the project.



Master Data should remain stable and reusable.



\---



\# Milestone



A major project objective consisting of multiple completed sprints.



\---



\# Project Owner



Person responsible for business decisions and repository maintenance.



\---



\# Reference Integrity



Verification that identifiers correctly reference existing records.



\---



\# Repository



The complete version-controlled project.



The repository is the primary source of truth.



\---



\# Repository Audit



Analysis of the current repository state performed before implementation.



\---



\# Session



One continuous AI collaboration session.



\---



\# Session State



The current implementation context recorded for future AI sessions.



\---



\# Sprint



A coherent package of work completed as one development cycle.



\---



\# Source of Truth



The authoritative source for a given decision.



The repository has the highest priority.



\---



\# Validation



The process of verifying repository consistency and data quality.



Validation evolves incrementally during project development.



\---



\# Validator



Software responsible for checking repository integrity and data correctness.



\---



\# Version



A specific state of the repository at a given point in time.



\---



\# Working Agreement



The collaboration contract between the Project Owner and the AI assistant.



\---
